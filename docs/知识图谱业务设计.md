# 知识图谱业务设计

> 基于实验室数据分析系统业务逻辑，以带材(Ribbon)为根节点构建知识图谱。
> 本文档覆盖外观特性、公式维护、判定等级、统计四条业务主线。

---

## 一、业务概述

知识图谱以 **带材(Ribbon)** 为根节点，描述检测中心从原始数据导入到最终统计判定的完整业务链路。

```
带材(Ribbon)
    ├─ 产品规格(ProductSpec) — 带材分为的产品类型
    │   ├─ 扩展属性(SpecAttribute) — 长度、层数、密度等
    │   └─ 判定规则(JudgmentRule) — 按规格维护的判定条件
    │           └─ 公式(Formula) — 判定依据的公式/指标
    ├─ 叠片数据(LaminationData) — 原始检测数据（22列检测列）
    ├─ 单片性能(SingleSheetPerf) — Ps铁损、Ss功率、Hc
    ├─ 外观特性(AppFeature) — 缺陷、等级、分类（可选）
    │   ├─ 特性大类(AppCategory)
    │   └─ 特性等级(AppLevel)
    └─ 报表配置(ReportConfig) — 统计汇总配置
```

## 四、核心业务逻辑

### 4.1 外观特性

- 带材导入叠片数据时**可能有外观特性，也可能没有**。
- 外观特性结构：**特性分类 → 特性名称 → 特性等级**。
- 数据库三张表：
  - `lab_appearance_feature_category`：特性大类
  - `lab_appearance_feature_level`：特性等级
  - `lab_appearance_feature`：具体特性项（关联大类和等级）

### 4.2 公式维护（lab_intermediate_data_formula）

- 路径：`/lab/intermediateDataFormula`
- `F_TABLE_NAME` 固定为 `INTERMEDIATE_DATA`
- `F_COLUMN_NAME` 对应 `IntermediateDataEntity` 属性名（如 `OneMeterWeight`、`Density`、`Labeling` 等）
- **三种公式类型**：

| 类型 | 枚举值 | 说明 |
|------|--------|------|
| CALC | 1 | 计算公式 — 导入过程计算后存入 `lab_intermediate_data` |
| JUDGE | 2 | 判定公式 — 需要在判定等级页面维护规则 |
| NO | 3 | 只展示 — 直接展示，不计算不判定 |

- **不区分产品规格**，所有产品共享同一套公式定义。

### 4.3 判定等级（lab_intermediate_data_judgment_level）

- 路径：`/lab/intermediateDataJudgmentLevel`
- **区分产品规格**（`F_PRODUCT_SPEC_ID`）
- `F_FORMULA_ID` 关联 `lab_intermediate_data_formula`，即判定哪一列
- 同一公式可维护多个判定等级，通过 **优先级(F_PRIORITY)** 控制先后顺序
- `F_CONDITION` 存储 JSON 条件，结构如下：

```json
{
  "groups": [
    {
      "id": "group-1",
      "name": "主条件组",
      "mode": "simple",
      "logic": "AND",
      "conditions": [
        {
          "id": "c1",
          "leftExpr": "[Width]",
          "operator": ">=",
          "rightValue": "119.5"
        }
      ]
    }
  ]
}
```

- 兜底判定：`F_IS_DEFAULT = true` 表示所有条件都不满足时使用。

### 4.4 统计

- 根据**判定列**进行统计，对应 `lab_intermediate_data` 的字段。
- 例如：**贴标** 对应 `lab_intermediate_data.F_LABELING`
- 叠片导入时已进行计算和判定，统计即对 `lab_intermediate_data` 某列做聚合（COUNT、GROUP BY 等）。
- 可看 `/lab/sub-table` 的**计算与判定详情日志**追踪过程。

## 五、数据库表映射

### 5.1 产品规格

```
lab_product_spec
    F_Id          → ProductSpec.id
    F_CODE        → ProductSpec.code
    F_NAME        → ProductSpec.name
    F_DETECTION_COLUMNS → ProductSpec.detectionColumns
    F_DESCRIPTION → ProductSpec.description
```

### 5.2 扩展属性

```
lab_product_spec_attribute
    F_PRODUCT_SPEC_ID → 关联 ProductSpec
    F_ATTRIBUTE_NAME  → SpecAttribute.name（长度、层数、密度）
    F_ATTRIBUTE_KEY   → SpecAttribute.key
    F_ATTRIBUTE_VALUE → SpecAttribute.value
    F_UNIT            → SpecAttribute.unit
    F_VERSION         → 版本号（关联 lab_product_spec_version）
```

### 5.3 公式维护

```
lab_intermediate_data_formula
    F_Id           → Formula.id
    F_COLUMN_NAME  → Formula.columnName（对应 IntermediateDataEntity 属性）
    F_FORMULA_NAME → Formula.name
    F_FORMULA      → Formula.expression
    F_FORMULA_TYPE → Formula.type（CALC/JUDGE/NO）
    F_UNIT_NAME    → Formula.unit
```

### 5.4 判定等级

```
lab_intermediate_data_judgment_level
    F_Id              → JudgmentRule.id
    F_FORMULA_ID      → 关联 Formula.id
    F_PRODUCT_SPEC_ID → 关联 ProductSpec.id
    F_NAME            → JudgmentRule.name（等级名称）
    F_QUALITY_STATUS  → JudgmentRule.qualityStatus
    F_PRIORITY        → JudgmentRule.priority
    F_CONDITION       → JudgmentRule.conditionJson
    F_IS_DEFAULT      → JudgmentRule.isDefault
```

### 5.5 外观特性

```
lab_appearance_feature_category → AppearanceCategory
lab_appearance_feature_level    → AppearanceLevel
lab_appearance_feature          → AppearanceFeature
    F_CATEGORY_ID      → BELONGS_TO_CATEGORY → AppearanceCategory
    F_SEVERITY_LEVEL_ID→ HAS_LEVEL → AppearanceLevel
```

## 六、前端渲染逻辑

### 6.1 布局方向

采用 **从左到右的业务流布局**：

```
[ Ribbon ] → [ ProductSpec ] → [ JudgmentRule ] → [ Formula ]
     ↓              ↓
[ LaminationData ]  [ SpecAttribute ]
[ SingleSheetPerf ]
[ AppearanceFeature ]
```
