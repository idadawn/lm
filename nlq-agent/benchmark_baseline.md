## Methodology change vs prior baseline

The previous baseline (2026-05-01) reported **40% error rate** at the same `c=10, n=50` settings. That number was an artifact of the benchmark setup, not a real application failure:

- The production middleware stack includes `RateLimitInMem` with default `max_requests=30, window_seconds=60` (per-IP).
- Under `httpx.ASGITransport`, all in-process requests share `request.client = None`, so they drain a single token bucket. The bucket holds 30 tokens, refills at 0.5/s, and the benchmark consumes 50 requests in ~0.05s — exactly 30 succeed (200) and 20 are short-circuited with 429.
- The remaining "throughput" number (977 req/s) was inflated because half the responses were rate-limit short-circuits, not real handler invocations.

The benchmark now patches `src.main.RateLimitInMem` with a passthrough middleware so it measures actual application throughput. Real rate-limit behaviour remains covered by `tests/unit/test_middleware.py`. See `scripts/benchmark.py:_PassthroughMiddleware` for the bypass implementation, and `tests/unit/test_benchmark_no_ratelimit.py` for the contract test.

## Reading the new numbers

- **0% error rate** — confirms the application path is healthy under concurrent SSE load with mocks.
- **338 req/s** — true throughput on the test machine. Lower than the prior inflated 977 because every request now exercises the full middleware stack + handler + SSE generation, not the rate-limit short-circuit.
- **p95 = 44ms** — typical for a 4-middleware FastAPI + SSE pipeline on this hardware.
- **p99 / p50 ratio = 2.6×** — modest tail amplification, healthy for ASGI workloads.
