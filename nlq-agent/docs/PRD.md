## 一、产品概述

### 1.1 背景与动机

现有系统（Poxiao Lab 模块）已在 MySQL 中构建了一套完整的**指标体系**，包含计算公式定义、等级判定规则和产品规格维度。质量工程师和生产管理人员在日常工作中需要频繁查询检测数据，但当前的交互方式依赖固定报表或手写 SQL，存在以下痛点：

一是**查询门槛高**，非技术人员无法自主获取数据，需依赖 IT 人员协助；二是**报表固化**，固定报表无法满足临时性、探索性的分析需求；三是**归因困难**，当某批次被判定为低等级时，用户难以快速理解原因；四是**洞察缺失**，系统不具备主动发现异常趋势的能力。

NLQ-Agent 旨在通过**自然语言交互**解决上述问题，让用户像与专家对话一样获取数据洞察。

### 1.2 产品定位

NLQ-Agent 是一款面向工业质量管理场景的 **AI 数据助手**，以"指标"为核心概念，以"智能体"为执行引擎，以"自然语言"为交互界面。它不是一个通用的 BI 工具，而是深度结合现有指标体系、公式定义和等级判定规则的**领域专属 AI 助手**。

### 1.3 目标用户

| 用户角色 | 典型使用场景 | 技术背景 |
|---------|------------|---------|
| **质量工程师** | 分析某批次被判低等级的原因，追溯问题根源 | 熟悉业务，不懂 SQL |
| **生产管理人员** | 查询班次/产线的指标达成情况，监控生产稳定性 | 熟悉业务，不懂 SQL |
| **质量总监** | 查看趋势报告，发现系统性问题，做决策支持 | 管理视角，需要结论性答案 |
| **工艺工程师** | 假设性分析，评估标准调整的影响 | 有一定技术背景，需要精确数据 |

## 二、核心功能需求

### 2.1 问答能力矩阵

系统支持四类问题，复杂度和 AI 推理深度依次递增：

#### 2.1.1 查询型（Query）

**定义**：用户希望获取某个指标在特定时间、班次、规格维度下的聚合值，并了解该值对应的等级。

**典型问法**：
- "1月甲班的 Ps 铁损平均值是多少？"
- "上个月叠片系数的最大值和最小值分别是多少？"
- "本周 120 规格的 A 级品率是多少？"

**AI 处理逻辑**：理解意图 → 识别指标/时间/规格实体 → 从指标元数据获取计算公式 → 生成并执行 SQL → 对照等级判定规则得出等级 → 选择合适图表 → 输出结果与计算说明。

**输出要求**：必须包含三部分——数值结果、计算方式说明（公式来源）、等级判定结论（含判定条件）。

#### 2.1.2 归因型（Root Cause）

**定义**：用户希望理解某条具体记录被判定为某等级的原因，尤其是为什么未能达到更高等级。

**典型问法**：
- "为什么炉号 1丙20260110-1 被判定为 B 级？"
- "这个炉号距离 A 级差在哪里？"
- "最近一周被判 C 级的炉号有哪些共同特征？"

**AI 处理逻辑**：按炉号查询该行所有原始数据 → 获取当前判定结果 → 对比目标等级（A 级）的所有判定条件 → 逐条标记满足/不满足 → 生成自然语言解释，指出关键差距。

**输出要求**：以"满足/不满足"对照表的形式展示各条件，并用自然语言总结核心原因，附带数值与阈值的直观对比图。

#### 2.1.3 洞察型（Insight）

**定义**：用户希望 AI 主动发现数据中的异常模式或趋势变化，而非仅回答固定问题。

**典型问法**：
- "最近三天的叠片系数有什么异常吗？"
- "本月哪个班次的铁损波动最大？"
- "近一个月 A 级品率的趋势是上升还是下降？"

**AI 处理逻辑**：获取指定时间段的时序数据 → 计算统计量（均值、标准差、趋势斜率）→ 运行异常检测算法（3σ 或 IQR）→ 将统计结论翻译为业务语言 → 给出可操作的建议。

**输出要求**：不仅给出数据，还要给出**结论性判断**（如"生产稳定性在下降"）和**可操作建议**（如"建议检查喷嘴状态"）。

#### 2.1.4 假设型（Hypothesis）

**定义**：用户希望评估改变某个判定标准或工艺参数后，对等级分布的影响。

**典型问法**：
- "如果把厚度极差的 A 级标准从 3.0 放宽到 3.5，A 级品比例会提升多少？"
- "如果 Ps 铁损的 B 级上限从 1.45 收紧到 1.40，B 级品中有多少会降为 C 级？"

**AI 处理逻辑**：获取近期批量原始数据到内存 → 在内存中修改判定条件 → 重新计算每条记录的等级 → 对比修改前后的等级分布 → 输出变化量和百分比。

**输出要求**：以"修改前 vs 修改后"的对比形式展示，包含合格率变化、各等级数量变化，以及受影响的具体炉号列表。

### 2.2 指标体系支持

系统基于 MySQL 中已定义的指标体系运作，支持以下指标类型：

| 指标类型 | 说明 | 示例 |
|---------|------|------|
| **直接读取型（NO）** | 直接展示原始字段值，无需计算 | PerfPsLoss（Ps 铁损）、PerfHc（矫顽力） |
| **计算型（CALC）** | 通过公式从原始字段计算得出 | 厚度极差 = MAX(Thickness1~N) - MIN(Thickness1~N) |
| **判定型（JUDGE）** | 根据等级规则判定质量等级 | 综合等级判定 |

所有指标均支持按**产品规格（ProductSpec）**区分判定标准，即同一指标值在不同规格下可能对应不同等级。

### 2.3 多轮对话

系统支持上下文感知的多轮对话，用户无需在每次追问时重复背景信息。上下文保留的实体包括：当前指标、当前时间范围、当前产品规格、当前炉号。

**典型多轮对话示例**：

> 用户：1月甲班的 Ps 铁损均值是多少？  
> AI：1月甲班 Ps 铁损均值为 1.23 W/kg，达到 **A 级**标准（≤1.30 W/kg）。  
>  
> 用户：那上个月呢？  
> AI：12月甲班 Ps 铁损均值为 1.28 W/kg，同样达到 **A 级**标准，但较1月上升了 0.05 W/kg。  
>  
> 用户：为什么12月的值更高？  
> AI：（切换为洞察型分析）12月甲班铁损均值偏高，主要集中在12月15日至20日期间，该时段标准差也明显增大……

### 2.4 图表自适应输出

AI 根据问题类型和数据特征自动选择图表形式，无需用户指定：

| 场景 | 图表类型 | 说明 |
|------|---------|------|
| 单指标时序趋势 | 折线图 | 含等级阈值参考线 |
| 等级分布占比 | 饼图 / 环形图 | 含各等级颜色编码 |
| 单值结果 | 仪表盘 / KPI 卡片 | 含等级色彩标注 |
| 多维度对比 | 分组柱状图 | 班次/产线/规格对比 |
| 异常检测结果 | 折线图 + 异常标注 | 异常点高亮显示 |
| 假设分析对比 | 对比柱状图 | 修改前后并排展示 |
| 归因对比 | 条形图 + 达标标注 | 各条件满足情况 |

### 2.5 计算透明性

每次 AI 回答必须包含**计算方式说明**，让用户理解数据的来源和计算逻辑。说明内容包括：指标的自然语言描述、计算公式（从 `Formula.Formula` 字段读取并翻译）、数据来源表和字段、等级判定条件（从 `JudgmentLevel.Condition` 读取并翻译）。

## 三、多端支持

### 3.1 Web 端

Web 端面向 PC 浏览器和平板设备，提供完整功能体验，包括：

**对话页面**：全屏对话界面，支持流式输出（打字机效果），消息中内嵌图表，支持图表交互（缩放、悬停提示）。

**指标看板页面**：以卡片形式展示核心指标的当前值和等级，支持时间范围筛选，点击指标卡片可跳转到对话页面进行深入分析。

**历史对话页面**：查看和恢复历史对话记录。

**系统配置页面**（管理员）：LLM 模型配置、指标元数据管理。

### 3.2 移动端（uni-app x）

移动端采用 **uni-app x** 开发，一套代码覆盖 Android、iOS、鸿蒙及微信小程序。移动端针对手机屏幕优化交互：

全屏对话模式，输入框固定在底部；图表支持手势缩放和横屏展示；回答卡片支持上下滑动展开详情；支持语音输入（调用系统语音识别）；离线时显示历史对话记录。

## 四、数据源支持

### 4.1 第一版：MySQL

第一版仅支持 MySQL 数据源，直接连接现有的 Poxiao Lab 数据库。系统使用**只读账号**访问，所有查询均经过 SQL 白名单校验，仅允许 `SELECT` 语句。

### 4.2 后续版本扩展

架构设计预留数据源扩展接口，后续版本可接入 PostgreSQL、ClickHouse、SQL Server 等数据库，以及 Excel/CSV 文件数据源。

## 五、LLM 模型支持

系统通过 **LiteLLM Proxy** 统一管理 LLM 接入，支持以下模型提供商：

| 接入方式 | 支持的模型 | 适用场景 |
|---------|----------|---------|
| **OpenAI API Key** | GPT-4.1、GPT-4o、o3 等 | 生产环境，高质量回答 |
| **OpenRouter** | Claude 3.5、Gemini 2.0、Llama 3.3 等 100+ 模型 | 灵活选择，成本优化 |
| **Ollama（本地）** | Qwen2.5、DeepSeek-R1、Llama 3.1 等 | 私有化部署，数据不出厂 |
| **vLLM（自托管）** | 任意 HuggingFace 模型 | 高性能私有化部署 |

用户可在界面的**模型选择器**中实时切换，系统自动检测所选模型是否支持 Function Calling（归因型、假设型问题的必要能力），不支持时自动降级提示。

## 六、用户界面设计规范

### 6.1 设计原则

**清晰透明**：每个 AI 回答都要展示推理过程和数据来源，让用户信服结论。

**渐进披露**：默认展示结论和关键数据，用户可展开查看详细推理过程和原始数据。

**工业风格**：配色以深色/中性色为主，等级颜色遵循行业惯例（绿色=优/A级，黄色=良/B级，红色=差/C级）。

**移动优先**：所有界面组件在 375px 宽度下均可正常使用。

### 7.2 开发路线图

**第一阶段（MVP，约 6 周）**

目标：交付查询型问答的完整闭环，验证核心架构可行性。

交付内容：FastAPI + LangGraph 基础框架搭建；语义层（指标元数据加载、时间解析、同义词匹配）；QueryAgent 全套工具（SQL 生成、执行、图表选择）；Web 端对话界面（Next.js + ai-elements）；基础多轮对话（时间/规格上下文复用）；LiteLLM 多模型接入；Docker Compose 本地开发环境。

**第二阶段（约 4 周）**

目标：交付归因型问答，上线指标看板。

交付内容：知识图谱构建（NetworkX，从 MySQL 加载）；RootCauseAgent 及其工具集；判定条件自然语言解释模块；指标看板页面（`@ant-design/charts`）；历史对话记录功能。

**第三阶段（约 5 周）**

目标：交付洞察型和假设型问答，完成移动端开发。

交付内容：InsightAgent（统计分析 + 异常检测）；HypothesisAgent（内存模拟计算）；uni-app x 移动端应用；移动端图表适配（xCharts）。

**第四阶段（持续迭代）**

扩展数据源支持；优化 LLM 提示词；建立用户反馈机制；接入 MCP 工具扩展能力；性能优化与监控完善。

## 八、非功能性需求

### 8.1 性能要求

| 指标 | 要求 |
|------|------|
| 查询型问题首字响应时间 | ≤ 3 秒 |
| 归因型问题首字响应时间 | ≤ 5 秒 |
| 流式输出速度 | ≥ 20 tokens/秒 |
| 图表渲染时间 | ≤ 1 秒 |
| 并发对话用户数 | ≥ 50 |

### 8.2 安全要求

所有 SQL 查询经过白名单校验，仅允许 `SELECT` 语句，禁止任何写操作。用户认证通过 JWT Token 实现，与现有 .NET 系统共享认证体系。敏感配置（数据库密码、API Key）通过环境变量注入，不写入代码仓库。

### 8.3 可观测性要求

所有 Agent 的执行过程通过 **LangSmith** 记录完整 Trace，包括工具调用参数、LLM 输入输出、执行耗时。异常情况触发告警通知。

## 九、验收标准

### 9.1 第一阶段验收标准

以下场景均能正确回答，且回答包含计算方式说明和等级判定：

1. "最近三天的 Ps 铁损平均值是多少？"
2. "1月甲班的叠片系数均值达到什么等级？"
3. "本月 120 规格的 A 级品率是多少？"（产品规格维度）
4. "那上个月呢？"（多轮追问）
5. 模型切换后上述问题仍能正确回答

### 9.2 第二阶段验收标准

1. "为什么炉号 1丙20260110-1 被判定为 B 级？"——能逐条列出满足/不满足的判定条件
2. 指标看板正确展示当前各指标值和等级
3. 点击看板上的"问 AI"按钮，能跳转到对话页面并自动带入上下文

### 9.3 第三阶段验收标准

1. "最近三天的叠片系数有什么异常吗？"——能给出统计结论和业务建议
2. "如果厚度极差放宽到 3.5，A 级品比例会提升多少？"——能给出修改前后的对比数据
3. 移动端 App 能完成上述所有查询，图表正常渲染

## 十、术语表

| 术语 | 定义 |
|------|------|
| **指标** | 对检测数据的度量维度，如"Ps 铁损"、"叠片系数"，对应 `LAB_INTERMEDIATE_DATA_FORMULA` 表中的一条记录 |
| **公式** | 指标的计算表达式，支持 EXCEL 语法和 MATH 语法，如 `AVG(RANGE(Thickness, 1, $DetectionColumns))` |
| **等级判定** | 根据指标值和阈值规则判定质量等级（如 A/B/C 级），规则存储在 `LAB_INTERMEDIATE_DATA_JUDGMENT_LEVEL` 表 |
| **产品规格** | 不同产品型号对应不同的等级判定标准，存储在 `LAB_PRODUCT_SPEC` 表 |
| **炉号** | 生产批次的唯一标识，格式为 `[产线][班次][日期]-[炉次]-[卷]-[分卷]` |
| **知识图谱** | 以图结构表达指标、公式、字段、等级、条件之间关系的内存数据结构 |
| **LLM 网关** | 统一管理多个 LLM 提供商接入的代理服务（LiteLLM Proxy） |
| **Agent** | 能够自主调用工具、执行多步推理的 AI 执行单元（基于 LangGraph） |
| **NLQ** | Natural Language Query，自然语言查询 |
