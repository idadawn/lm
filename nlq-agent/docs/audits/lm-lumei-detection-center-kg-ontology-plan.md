# 山东路美磁性材料检测中心知识图谱本体与“小美”问答助手方案

作者：**Manus AI**  
适用项目：`idadawn/lm`  
业务对象：山东路美磁性材料有限公司检测中心  
核心场景：非晶带材检测、数据录入、检测明细生成、规格判定、指标统计与问答助手“小美”

## 1. 业务理解与系统边界

山东路美磁性材料有限公司检测中心的业务核心不是一个通用知识库，而是围绕**非晶带材检测业务**形成的结构化数据、计算规则、产品规格、判定规则与统计分析能力。知识图谱应该服务于两个目标：第一，让业务人员能够从“带材”出发理解某一批次、某一卷或某一规格产品的检测依据；第二，让问答助手“小美”能够基于图谱先回答“是什么、从哪里来、怎么算、怎么判定”，再进一步调用统计分析或明细数据回答“是否合格、为什么异常、哪些指标贡献最大”。

当前系统中，已实现的数据入口包括**叠片数据导入** `/lab/raw-data`、**单片性能录入** `/lab/magneticData`、**带材检测明细表** `/lab/sub-table`。**环样性能录入**属于新需求，尚未实现。基于这些入口，知识图谱不应再把所有“规格、规则、公式、属性”平铺成散乱节点，而应以业务主对象“带材”为根节点，围绕数据流和判定流组织图谱。

| 业务模块 | 当前状态 | 图谱中的角色 | 说明 |
|---|---:|---|---|
| 叠片数据导入 `/lab/raw-data` | 已做 | 原始检测数据源 | 公式计算的主要输入之一，最终生成或补充带材检测明细表。 |
| 单片性能录入 `/lab/magneticData` | 已做 | 性能检测数据源 | 反映单片样品性能，可参与规格判定、性能追踪和统计分析。 |
| 环样性能录入 | 新需求未做 | 待接入性能数据源 | 应作为与单片性能并列的数据类型，后续接入图谱。 |
| 带材检测明细表 `/lab/sub-table` | 已做 | 检测事实表/结果表 | 承接原始数据、公式计算结果和判定结果，是问答与统计的重要事实来源。 |
| 产品规格 | 已有概念 | 判定上下文 | 判定规则区分产品规格，同一指标在不同规格下的阈值可能不同。 |
| 带材外观特性 | 已有概念 | 质量表征 | 应作为带材根节点下的特性集合，而不是无差别散点节点。 |
| 公式 | 已有概念 | 计算规则 | 叠片数据导入后通过公式维护计算得到检测明细表字段。 |
| 指标判定 | 已有概念 | 统计与质量判定规则 | 主要用于统计计算，也可解释某指标是否达标。 |

## 2. 以“带材”为根节点的知识图谱本体设计

新的知识图谱应从“数据库表驱动”转向“业务对象驱动”。根节点应为**带材**，它代表一次检测业务中被检测、被计算、被判定和被统计的核心对象。围绕带材展开的一级关系包括：属于某个产品规格、具有外观特性、产生叠片数据、产生单片性能、产生环样性能、生成检测明细、适用公式、适用判定规则。

> **建模原则**：产品规格决定判定上下文，公式决定检测明细的计算来源，指标判定决定统计口径与质量解释，检测明细表承载最终事实。带材是所有事实、规则与解释的汇聚点。

| 节点类型 | 建议英文标识 | 关键属性 | 是否默认展示 | 说明 |
|---|---|---|---:|---|
| 带材 | `Ribbon` | 带材编号、批次、卷号、生产日期、规格编码、检测状态 | 是 | 图谱根节点，所有查询应尽量先定位到带材。 |
| 产品规格 | `ProductSpec` | 规格编码、规格名称、宽度、厚度、牌号、适用客户/产线 | 是 | 判定规则按规格区分。 |
| 外观特性 | `AppearanceFeature` | 特性名称、等级、位置、描述、是否缺陷 | 是，但建议聚合 | 如边裂、脆边、亮线等。 |
| 叠片数据 | `LaminationData` | 导入批次、原始字段、采样位置、导入时间 | 按需展示 | `/lab/raw-data` 来源，主要是公式输入。 |
| 单片性能 | `SingleSheetPerformance` | 性能指标、测量值、单位、样品编号 | 按需展示 | `/lab/magneticData` 来源。 |
| 环样性能 | `RingSamplePerformance` | 性能指标、测量值、单位、环样编号 | 按需展示 | 新需求，建议与单片性能统一抽象为性能数据。 |
| 检测明细 | `DetectionDetail` | 明细编号、指标值、计算值、判定结果、数据来源 | 是 | `/lab/sub-table`，是事实结果层。 |
| 公式 | `Formula` | 公式编码、公式名称、表达式、输入字段、输出字段、版本 | 是 | 用于解释“这个字段怎么算出来”。 |
| 判定规则 | `JudgementRule` | 规则编码、指标、上下限、规格、单位、判定逻辑、版本 | 是 | 产品规格相关。 |
| 指标 | `Metric` | 指标编码、指标名称、单位、统计口径 | 是 | 连接公式、检测明细与判定规则。 |
| 判定结果 | `JudgementResult` | 合格/不合格/预警、原因、命中规则、时间 | 是 | 可作为检测明细的结果属性，也可独立为节点。 |

## 3. 推荐关系模型

关系比节点更重要。当前页面混乱的核心原因之一，是没有清楚表达“数据如何流动、规则如何命中、结果如何产生”。建议把关系分为四类：对象归属关系、数据来源关系、计算关系、判定关系。

| 起点 | 关系 | 终点 | 业务含义 | 图谱展示建议 |
|---|---|---|---|---|
| 带材 | `BELONGS_TO_SPEC` | 产品规格 | 该带材对应哪个产品规格 | 默认展示 |
| 带材 | `HAS_APPEARANCE` | 外观特性 | 该带材有哪些外观特性或缺陷 | 默认聚合展示 |
| 带材 | `HAS_LAMINATION_DATA` | 叠片数据 | 带材对应的叠片数据导入记录 | 默认折叠 |
| 带材 | `HAS_SINGLE_SHEET_PERF` | 单片性能 | 带材对应的单片性能记录 | 默认折叠 |
| 带材 | `HAS_RING_SAMPLE_PERF` | 环样性能 | 带材对应的环样性能记录 | 新需求接入后展示 |
| 叠片数据 | `CALCULATED_BY` | 公式 | 原始数据通过公式计算 | 点击公式时展示 |
| 公式 | `OUTPUTS_METRIC` | 指标 | 公式输出哪个检测指标 | 默认展示 |
| 指标 | `RECORDED_IN` | 检测明细 | 指标值落到检测明细表 | 默认展示 |
| 产品规格 | `USES_RULE` | 判定规则 | 规格使用哪些判定规则 | 默认展示 |
| 判定规则 | `JUDGES_METRIC` | 指标 | 规则判定哪个指标 | 默认展示 |
| 检测明细 | `HIT_RULE` | 判定规则 | 明细值命中哪条规则 | 选中明细时展示 |
| 检测明细 | `HAS_RESULT` | 判定结果 | 最终是否合格或预警 | 默认展示 |

## 4. 图谱主视图应该如何展示

图谱页面不应默认显示全量节点。建议重构为“**带材中心图谱 + 右侧解释面板 + 顶部问答入口**”。默认视图以某个带材为中心，左侧表示数据来源，中间表示带材与检测明细，右侧表示公式、指标与判定规则。这样既符合业务流，也符合用户理解路径。

```text
叠片数据       ┐
单片性能       ├──> 带材 ──> 检测明细 ──> 指标 ──> 判定规则
环样性能       ┘       │          ↑          │        ↑
                       │          │          │        │
                       └──> 产品规格 ────────┘        │
                       └──> 外观特性                  │
公式 ───────────────────────────────> 指标 ───────────┘
```

| 区域 | 展示内容 | 交互方式 | 目的 |
|---|---|---|---|
| 中央画布 | 带材、规格、检测明细、指标、规则、公式 | 点击节点、展开一跳、路径高亮 | 理解业务链路。 |
| 左侧筛选 | 带材编号、批次、规格、检测日期、数据类型 | 搜索与过滤 | 避免全量图造成混乱。 |
| 右侧详情 | 节点详情、数据来源、公式表达式、判定规则、命中原因 | 随选中节点切换 | 替代在图上显示所有属性。 |
| 顶部问答 | “问小美”输入框 | 自然语言查询 | 将图谱与问答助手联动。 |
| 底部图例 | 节点类型、关系类型、判定颜色 | 固定说明 | 降低学习成本。 |

## 5. “小美”问答助手的业务能力边界

“小美”不应该只是把 LLM 接到数据库上，而应该优先走知识图谱路径。对于检测中心业务，典型问题可以分为对象查询、计算解释、判定解释、统计分析和异常原因分析五类。

| 问题类型 | 示例问题 | 小美应走的路径 | 回答结构 |
|---|---|---|---|
| 对象查询 | “炉号 1丙20260110-1 是什么规格？” | 带材 → 产品规格 | 先给结论，再列规格字段和数据来源。 |
| 外观查询 | “这卷带材有哪些外观问题？” | 带材 → 外观特性 | 按特性、等级、位置汇总。 |
| 公式解释 | “检测明细里的某指标是怎么算的？” | 检测明细 → 指标 → 公式 → 输入字段 | 展示公式表达式、输入字段、输出字段。 |
| 判定解释 | “为什么判为 C 级？” | 带材 → 规格 → 判定规则 → 指标 → 检测明细 | 展示命中规则、实际值、阈值、判定结果。 |
| 统计分析 | “最近一周 B 级占比是多少？” | 图谱定位指标/规则 → 查询明细表统计 | 给统计口径、样本量、比例和时间范围。 |
| 根因分析 | “为什么这个批次不合格多？” | 图谱定位规则与指标 → 统计异常指标 → 关联外观/性能数据 | 先给可能原因，再给证据链。 |

建议小美的回答遵循固定模板：**结论、依据、计算/判定过程、关联数据、可继续追问**。例如：

> 小美回答示例：  
> “这条带材当前匹配的产品规格是 **A规格**。判定为 **C级** 的主要原因是指标 `后部最大值` 命中了该规格下的 C 级判定规则。实际值为 `x`，规则阈值为 `y-z`，数据来源于 `/lab/sub-table` 的检测明细记录，相关原始数据来自 `/lab/raw-data` 的叠片导入批次。你可以继续问我：‘这个指标是怎么算出来的？’ 或 ‘同规格最近一周 C 级占比是多少？’”

## 6. 环样性能录入的新需求建模建议

环样性能不应作为孤立页面单独存在，而应与单片性能一起归入“性能数据”本体。可以在数据库和图谱中抽象一个上层概念 `PerformanceRecord`，再用 `performance_type` 区分 `single_sheet` 与 `ring_sample`。如果业务字段差异较大，也可以保留两个节点类型，但在问答层统一称为“性能数据”。

| 设计项 | 推荐做法 | 原因 |
|---|---|---|
| 页面路径 | `/lab/ring-sample` 或 `/lab/ringSampleData` | 与 `/lab/magneticData` 保持命名一致。 |
| 图谱节点 | `RingSamplePerformance` | 与单片性能并列，便于解释。 |
| 公共抽象 | `PerformanceRecord` | 便于小美统一回答性能相关问题。 |
| 与带材关系 | `HAS_RING_SAMPLE_PERF` | 从带材根节点展开。 |
| 与指标关系 | `MEASURES_METRIC` | 表示该性能记录测量哪个指标。 |
| 与判定规则关系 | 通过 `Metric` 间接连接 | 避免规则直接绑死某类数据。 |

## 7. 数据落地建议

从落地角度看，建议不要一开始追求复杂 Neo4j 全量同步，而是先建立一个清晰的“图谱视图层”。也就是说，后端从现有业务表中提取节点和边，按前端需要返回某个带材的一跳或两跳子图。等业务稳定后，再考虑把图谱持久化到图数据库。

| 接口 | 建议用途 | 返回内容 |
|---|---|---|
| `GET /api/kg/ribbon/search` | 搜索带材 | 带材编号、规格、批次、检测状态。 |
| `GET /api/kg/ribbon/{id}/subgraph?depth=1` | 查询某带材局部图谱 | 带材、规格、外观、检测明细、核心指标。 |
| `GET /api/kg/ribbon/{id}/explain/judgement` | 判定解释 | 命中规则、实际值、阈值、结果、来源。 |
| `GET /api/kg/metric/{id}/formula-lineage` | 公式血缘 | 输入字段、公式、输出指标、明细字段。 |
| `POST /api/kg/ask` | 小美图谱问答 | 解析意图、图谱路径、回答、证据链。 |

## 8. 前端组件建议

对于当前 `/lab/knowledge-graph` 页面，建议短期不要再展示全量图，而是改成“搜索带材后展示局部图谱”。如果继续使用 G6，应采用分层布局；如果希望最快改善观感，可使用 `relation-graph-vue3`。无论使用哪个组件，都应遵守相同展示原则：属性进详情面板，图上只放核心业务对象；边标签默认隐藏；点击节点显示一跳邻居；搜索只加载局部子图。

| 场景 | 推荐方案 | 说明 |
|---|---|---|
| 快速做出业务可读图谱 | `relation-graph-vue3` | 更接近业务关系图，Vue 上手快。 |
| 保留现有技术栈并增强交互 | `@antv/g6` | 需要重写布局、节点、边、交互。 |
| 大规模图谱浏览 | `sigma.js` | 适合几千节点，但业务节点样式成本高。 |
| 复杂图算法与路径分析 | `cytoscape.js` | 适合后续做路径、社区、复杂子图分析。 |

## 10. 结论

基于你补充的业务背景，lm 项目的知识图谱应该从“规则/规格/公式的可视化集合”升级为“**带材检测业务本体**”。真正的根节点是**带材**，不是产品规格，也不是指标。产品规格用于确定判定上下文，公式用于解释检测明细如何由叠片数据计算得到，指标判定用于统计和质量解释，检测明细表则是最终事实承载层。

“小美”问答助手的核心价值应是把这些链路讲清楚：某条带材是什么规格，录入了哪些数据，哪些字段由公式计算，哪些指标命中了哪条规则，为什么得到当前判定，以及同类规格在统计上是否存在异常。只有这样，知识图谱和问答助手才会真正服务检测中心业务，而不是停留在一张难看的关系图上。
